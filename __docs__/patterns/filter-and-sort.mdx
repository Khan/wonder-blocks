## The hierarchical logic of discovery

### Logic framework: how they play together

Regardless of which components are selected for your specific use case, the system must follow these foundational rules:
Additive filtering (AND): When a user selects a value from a Dropdown and then types in the Search Bar, the results must reflect both. The search is conducted within the already filtered subset. 
Inclusive selection (OR): Within a single component—specifically Checkboxes—selecting multiple items should expand the results (e.g., "Show me Blue OR Red items"). 
Cascading dependencies: Primary filters (e.g., School) dynamically "prune" secondary filters (e.g., Grade) to show only valid, overlapping options. If a primary filter is changed, any now-invalid secondary selections are cleared to prevent "No Results" states. 
Global reset: Provide a "Clear filters" function as soon as a single selection is made. This ensures users can return to a neutral state instantly without manual deselection, regardless of how many components are currently in use.

This section illustrates the logic framework in practice, transitioning from broad keyword discovery to granular category refinement. ### Filter bar

The horizontal filter bar is located directly above the content list to provide immediate access to data refinement tools.
It can feature things like (and in this order) Search field for keyword queries, embedded dropdowns for high-priority + preset selections (e.g., Last 30 days), compact additive dropdowns for common categories (e.g. School and Grade), and for advanced segments, the More Filters button opens a comprehensive modal for less frequent criteria.

<img src={theHierarchicalLogicOfDiscovery2Img} alt="The hierarchical logic of discovery 2" />

### Search field

A search field allowing users to type keywords to find specific items within the dataset. As you type, the system provides a list of auto-suggestions—such as specific class names—to help you find what you need faster. Once you select a suggestion or press enter, the main results list will refresh to display your matches.

### Embedded filter dropdown (single-select)

This variation is used for essential criteria that require a value to be active at all times, such as Date. Typically restricted to a single selection at a time to maintain a clear, focused data view. Because these are fundamental to the dataset, they remain visible in the filter bar as the primary anchors for the results displayed. Only use this for high-level "View" toggles (e.g., Date Range or Department) where the system cannot show "All" data without a primary category being selected.
Use them sparingly (1–2 max) to avoid over-constraining the data and forcing a "Zero Results" state before discovery begins.

### Compact additive dropdown (single or multi-select)

This specific style of dropdown is designed for high-frequency use within the horizontal filter bar. It allows users to manage common data categories—such as School or Grade—efficiently within a limited space.
Limit to 4 max of these categories to prevent UI clutter and ensure the most common data segments remain accessible.

### More filters modal

The More filters component serves as a secondary control center for complex or less frequent filtering needs, ensuring the primary interface remains uncluttered.
