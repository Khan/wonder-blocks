# Borders, radius, lines

<img src={bordersRadiusLinesImg} alt="Borders, radius, lines" />

### Borders, radius, lines

The structural details that define edges, shape personality, and signal separation. Used consistently, they're invisible — users feel the system without noticing it. Used inconsistently, they become visual noise that undermines trust in the design.

## Borders

A border makes an edge explicit. It says: this is where something begins or ends. Use a border when that boundary carries meaning — when a container needs to be visibly distinct from what surrounds it, or when an element's interactive affordance needs to be communicated at a glance.

### Principles

Structural

### Borders define containment

A border claims an edge. When everything has one, the signal disappears — reserve borders for contexts where the boundary is meaningful.

Accessible

### Focus borders are non-negotiable

Focus rings show keyboard users where they are. They must be high-contrast and visible — never suppressed without a fully accessible replacement.

### When to use a border

Form inputs, selects, and text areas — the border communicates that the field is interactive and editable

Cards or containers that sit on the same background color as the page — when there's no contrast difference to define the edge, a border does that job

Selected or active states where the border color changes to communicate state

Any element that needs to feel distinct from its surroundings without using elevation

### When not to use a border

On elements that already sit on a contrasting background — the contrast creates the boundary, the border is redundant

As a decorative device — if the border isn't telling the user something about the edge of the element, it shouldn't be there

### Color and weight

Border color and width always come from the token system — never raw hex values or arbitrary pixel values. Border widths use fixed pixel units (not rems) and do not scale with zoom. This is intentional. Token reference lives in Figma.

## Radius

Radius is one of the most visible personality signals in a design system. The degree of rounding on a component communicates where it sits in the hierarchy — small, dense components like badges and chips sit at one end; large containers like modals and cards sit at the other. Consistent radius is what makes a collection of components feel like a system.

### Principles

Hierarchical

### Radius follows component scale

Smaller components carry less radius; larger components carry more. Applying one value to everything flattens the hierarchy radius is meant to reinforce.

Relational

### Nested radius must be calculated, not guessed

When a rounded container holds a rounded child, the inner radius should equal the outer radius minus the gap between them. If both use the same value, the shapes look unrelated rather than nested.

### When to use radius

On interactive components — buttons, inputs, chips, badges, tags

On containers that hold content — cards, modals, drawers, tooltips, popovers

On any element where the degree of rounding should reflect its scale in the hierarchy

### When not to use radius

On full-bleed elements that extend to the edge of a screen or container — sharp corners are correct here

On an inner element when the calculated inner radius would be zero or negative — use sharp corners instead

Inconsistently — if the same component type uses different radius values in different contexts, the system feels arbitrary

### Radius values

Radius tokens follow the same sizing scale as the rest of the system. Token reference lives in Figma.

## Lines

A line is the minimum possible separator — 1px, a subtle color, nothing more. Lines communicate that two things on either side of them are distinct. The problem is that lines are easy to reach for and easy to overuse. Before adding a line, ask whether spacing alone would do the same job.

### Principles

Restrained

### Spacing first, lines second

A line should only appear when spacing alone isn't enough. In most layouts, spacing handles separation without any lines at all — lines become necessary only in dense contexts where items are too compressed for spacing to do the job.

Structural

### Lines are not headings

A heading names a section; a line marks a boundary. When used together, pick one — using both is redundant.

### When to use a line

Between list items in dense lists where spacing alone doesn't create enough separation

Between major sections on surfaces where a hard boundary is more appropriate than spacing alone

In data tables, to separate rows, columns, or both — using one consistent strategy throughout

### When not to use a line

When spacing is already doing the job — adding a line on top of adequate spacing is redundant

Directly below a section heading — the heading already marks the boundary; the line adds nothing

As decoration — a line that isn't separating two distinct things shouldn't be there

### Horizontal dividers

Use horizontal dividers between list items in dense lists where spacing alone doesn't provide enough separation, and between major sections on surfaces where a hard boundary is more appropriate than spacing. Dividers should use a subtle border color — a line that draws attention to itself has failed.

## Usage guidelines

### Always use semantic tokens for color

Border and line colors should always come from the semantic token system. Never use raw hex values or primitive tokens — semantic tokens ensure borders respond correctly to theme changes and meet contrast requirements.

<img src={usageGuidelines1Img} alt="Usage guidelines 1" />

### Don't use borders and shadows together

Both communicate containment. On the same element, they're redundant. If a card or panel needs to feel elevated, use shadow. If it needs to sit flat with a visible edge, use a border.

<img src={usageGuidelines2Img} alt="Usage guidelines 2" />

### Respect nested radius

When a component with rounded corners contains a child with rounded corners, calculate the inner radius rather than copying the outer value. Inner radius = outer radius − gap between them. If the calculation results in zero or a negative number, use sharp corners on the inner element.

<img src={usageGuidelines3Img} alt="Usage guidelines 3" />

### Suppress borders before suppressing focus rings

Focus rings may be visually adjusted — color, offset, width — but never removed without a fully accessible replacement. A missing focus ring is an accessibility failure, not a design preference.

<img src={usageGuidelines4Img} alt="Usage guidelines 4" />

### Lines should be lighter than you think

A border color that reads as "obvious" in isolation often reads as "heavy" in context. Err toward subtlety — lines communicate separation, not emphasis.

<img src={usageGuidelines5Img} alt="Usage guidelines 5" />

## Accessibility

### Focus rings must meet contrast requirements

The focus ring must have a contrast ratio of at least 3:1 against the adjacent colors. This applies in both light and dark mode. Do not use low-contrast or thin focus styles as a way to make focus states less visible — users who rely on keyboard navigation depend on them.

### Borders cannot be the only signal for required fields or error states

Color alone is not sufficient to communicate state. A red border on an error field must be accompanied by an error message, an icon, or another non-color indicator. Users with color vision deficiencies need a redundant signal.

### Ensure dividers are not mistaken for interactive elements

A horizontal rule should not be styled in a way that could be confused with a button, link, or selectable row. Keep lines visually subdued and consistent with their role as structural separators.
