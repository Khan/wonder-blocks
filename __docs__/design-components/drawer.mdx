# Drawer

<div style={{display: "flex", gap: "8px", marginBottom: "8px"}}>
  <NeutralBadge label="Contextual reference/Side-task" />
</div>

A secondary surface that slides in from the right edge. It allows users to view supplemental data or perform quick sub-tasks while maintaining a visual "anchor" to the primary page content.

<img src={drawerImg} alt="Drawer" />

## When to use

The choice to use a drawer is based on when the user needs to "glance" or "pivot" without full interruption.

| Use when | Rule | Example use |
| --- | --- | --- |
| Keeping place in a long- form activity | Use when the user needs to reference something without losing their current context. | Checking the District Leaderboard without losing your spot in a daily streak. |
| Showing content that  supports the current view | Use to surface supplemental content that enriches — but doesn't replace — what's on screen. | Opening Lessons from Practice mode to review a concept before answering. |
| Checking status that  doesn't require data entry | Use for high-frequency check-ins where the user reads, not writes. | Viewing a detailed Gem achievement count or point breakdown. |

## When NOT to use

| Don’t use when | Recommended alternative | Rationale |
| --- | --- | --- |
| The user needs to make  a high-stakes or  creative decision | Modal | If checking your gem count turns into customizing your Khanmigo, then the  drawer is too cramped. High-fidelity creative tasks need a full-page canvas. |
| The secondary task  becomes the primary task | Full page | If "Viewing an assignment" turns into "Doing the assignment," the drawer is too cramped. Move to a full-page view. |
| The information is a single  value or definition | Tooltip / popover | If the information is just a single definition or a number, a drawer is overkill. |

## Interaction

### Opening and entry

- ✅ Always slides in from the right edge. The slide direction is the spatial cue that content is supplemental, not a replacement.
- ✅ Opens immediately on a single selection — no hover states, no delay, no confirmation step.

### Dismissal

- ✅ Users exit via the × button in the header, clicking the scrim, or pressing Esc.
- ❌ Do not close the drawer automatically after a timer or inactivity. The user controls dismissal.

### Background locking

- ✅ Background is locked while the drawer is open to prevent accidental scrolling of the main page.

### Navigation from the drawer

- ✅ If the drawer contains a link to a full page (e.g. "Go to achievements"), clicking it closes the drawer and navigates the main page.
- ❌ Never load a new URL inside the drawer body. It signals a page — use a full-page transition instead.
- ❌ Never open a drawer from within a drawer. Nested drawers create disorienting spatial contexts.

## Placement

Drawers should feel like a natural extension of the right sidebar — same surface color, same typographic scale, same horizontal margins.

- ✅ Sticky header: The title and × remain locked at the top at all times.
- ✅ Sticky footer: Any primary actions (e.g. "Customize Khanmigo") should remain fixed at the bottom. If the content (like a long leaderboard) requires scrolling, the footer must not scroll with it.
- ✅ Internal scroll: Only the middle body area scrolls if the content exceeds the screen height.
- ❌ Do not use nested scroll containers within the body. Flatten the content hierarchy instead.

## Copy

### Headline

Keep it to 1–3 words. Use a noun phrase that describes the content — not the action.

<img src={copy1Img} alt="Copy 1" />

### Action labels

Use clear, directional verbs. If an action leads to a full page, use the destination name.

<img src={copy2Img} alt="Copy 2" />

### Body content

Should be easily scannable. Use data, labels, and short summaries. Not paragraphs. If the content needs paragraphs to make sense, it belongs on a full page.
